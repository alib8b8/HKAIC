## 问题 #1: P2P网络层无真实Socket通信

### 问题描述
原代码 `p2p_network.py` 中的 `P2P网络` 类完全是进程内模拟，没有实际的TCP/UDP Socket通信。所有P2P方法（握手、广播、心跳）都在内存中执行，无法跨进程通信。

### 根因
项目从单进程单元测试演进，P2P层设计为逻辑模型而非真实网络层。核心业务逻辑（共识、账本、交易）正确，但缺少网络I/O绑定。

### 修复方案
新建 `scripts/node_process.py`，添加基于 `http.server.HTTPServer` 的真实RPC服务器和基于 `urllib.request` 的P2P客户端：
- 每个节点进程运行独立的HTTP服务器（端口8841-8845）
- P2P通信通过HTTP POST/GET实现
- 支持 `/p2p/ping`、`/p2p/register`、`/p2p/block`、`/p2p/tx` 等端点

### 影响文件
- 新增: `scripts/node_process.py`（523行）
- 未修改原代码

## 问题 #2: 首次出块失败 — 交易池为空

### 问题描述
启动节点后，创建钱包并铸币，但 `force_block` 返回"无交易可出块"。

### 根因
铸币操作 `ledger.铸币()` 直接修改账本余额，不经过交易池（`tx_engine.交易池`）。区块链的 `出块()` 方法检查 `self._待确认交易` 是否为空，而交易池中没有任何待确认交易。

### 修复方案
在 `铸币给钱包()` 方法中，同时向交易引擎提交一笔铸币交易：
```python
def 铸币给钱包(self, 钱包名, 金额_HKAIC):
    self.ledger.铸币(w.地址, 金额_鸿坤)
    # 同时添加铸币交易到交易池
    self.tx_engine.创建转账("MINT", w.地址, 金额_鸿坤, 0, 交易优先级.高)
```

### 影响文件
- 修改: `scripts/node_process.py` — `铸币给钱包()` 方法

## 问题 #3: 共识引擎推进后未选出出块者

### 问题描述
调用 `consensus.推进共识()` 返回 `None`，导致无法出块。

### 根因
`推进共识()` 内部调用 `epoch_mgr.推进阶段()`，每调用一次只推进一个阶段（提案→预投票→预提交→提交）。只调用一次时，从提案阶段推进到预投票阶段，还未到达提交阶段（出块者选举阶段）。

### 修复方案
测试网采用宽松模式：
1. 先调用 `推进共识()` 尝试正常流程
2. 如果返回None或超时，直接调用 `判定出块权()` 用PoEI公式选出一个出块者
3. 如果仍然没有出块者，当前节点自己出块（测试网宽松模式）

```python
出块者 = self.consensus.推进共识()
if 出块者 is None or "超时" in str(出块者):
    候选 = list(self.consensus._S.keys())
    出块者 = self.consensus.判定出块权(候选, epoch.种子)
if 出块者 is None:
    出块者 = self.validator_name  # 测试网回退
```

### 影响文件
- 修改: `scripts/node_process.py` — `尝试出块()` 方法

## 问题 #4: 出块确认时余额重复扣减

### 问题描述
转账交易在 `tx/send` 接口中已经通过 `ledger.转账()` 执行了余额变更，但 `尝试出块()` 的确认逻辑又会再次调用 `ledger.转账()`，导致余额被扣两次。

### 根因
交易执行和交易确认是两个独立流程，但两者都调用了 `ledger.转账()`。原设计是交易只进交易池，确认时才执行转账。但为了简化测试网，`tx/send` 接口直接执行了转账。

### 修复方案
1. 铸币交易（发送地址为"MINT"）在出块确认时跳过账本操作
2. 普通转账交易在 `tx/send` 时已执行，出块确认时跳过
3. 生产环境应改为：交易只进交易池，出块确认时才执行

```python
if result["发送"] == "MINT":
    self.stats["tx_processed"] += 1
    continue  # 铸币交易已在账本中
```

### 影响文件
- 修改: `scripts/node_process.py` — `尝试出块()` 方法中的确认逻辑
